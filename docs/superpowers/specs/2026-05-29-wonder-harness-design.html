원더 하네스 (wonder-harness) 설계 스펙

📅 2026-05-29 👤 InHo Jeong <inho.jeong@wonderit.kr> 📦 plugins/wonder-harness 🏷️ v0.1.0 → 다음 마이너
요약 — 원더 하네스는 오케스트레이션 파이프라인 방식의 Claude Code 개발 하네스 플러그인이다. 사용자가 작성한 요청 문서를 입력으로, planner → templater → developer → ruler 4개 전문 에이전트를 슬래시 커맨드가 순차 구동한다. 템플릿 탐색을 강제하는 하드 차단 훅과, ruler가 소유하는 스택 고정형(Spring Boot + JSP/jQuery/Thymeleaf) 규칙 4종이 일관성을 보장한다. 대상은 wonderit 팀 전용.

1작동 모델 — 오케스트레이션 파이프라인

요청 문서가 파이프라인의 진입점이다. 메인 루프가 에이전트를 직접 호출할 수 없으므로, 슬래시 커맨드가 4개 에이전트를 순차 디스패치한다.

create_request.md─▶ /wh-create─▶ planner─▶ templater─▶ developer─▶ ruler

/wh-modify 는 modify_request.md 로 동일 파이프라인을 modify 모드로 구동. /wh-review 는 기존 산출물을 단독 검토.

2컴포넌트 인벤토리

분류항목위치상태
커맨드wh-create, wh-modify, wh-reviewcommands/신규
에이전트planner, templater, developer, ruleragents/신규
훅템플릿 탐색 강제 (PreToolUse + PostToolUse)hooks/hooks.json신규
규칙backend, frontend, security, template-metarules/신규
스킬grill-me, handoff, write-a-skillskills/존재
요청 양식create_request.md, modify_request.md플러그인 시드 → .claude/requests/신규
템플릿 시드Spring/JSP 범용 스캘폴드 + index 스키마플러그인 시드 → .claude/templates/신규

3커맨드 — 파이프라인 진입점

원래 구성요소 목록엔 커맨드가 없었으나, 파이프라인 구동을 위해 필수로 추가한다.

커맨드입력동작
/wh-create.claude/requests/create_request.md검증 게이트 통과 시 planner→templater→developer→ruler 순차 디스패치 (create 모드)
/wh-modify.claude/requests/modify_request.md검증 게이트 통과 시 동일 파이프라인 (modify 모드)
/wh-review대상 경로(인수)기존 코드/템플릿/규칙을 해당 에이전트로 단독 검토
검증 게이트 (fail-fast). /wh-create·/wh-modify 는 파이프라인 시작 전 요청 문서의 필수 섹션 충족 여부를 검사한다. 누락 시 파이프라인을 중단하고 누락 항목 목록을 보고한다. 불완전한 입력으로 파이프라인이 진입하지 않도록 경계에서 검증한다.

4에이전트 4종

각 에이전트는 단일 <name>.md 파일이며, 시스템 프롬프트가 create / modify / review 3모드를 핸들러로 전달받은 작업에 따라 내부 분기한다.

planner

모듈 생성 계획 수립·수정·검토. 요청 문서를 해석해 모듈 분해·의존성·구현 순서를 산출.

templater

템플릿 생성·수정·검토. template-meta 규칙 준수. .claude/templates/index.json 갱신·색인.

developer

코드 생성·수정·검토. Spring Boot(Java/Kotlin) + JSP/jQuery/Thymeleaf/그리드 스택 인지. 템플릿 우선 활용.

ruler

규칙 생성·수정·검토 + 파이프라인 종단 규칙 검증. 규칙 4종의 소유자.

5강제 훅 — 템플릿 탐색 하드 차단

Node.js 내장 모듈만 사용하는 경량 결정론적 훅. 두 이벤트가 협력해 "탐색 여부" 상태를 추적한다.

  1. PostToolUse(Read) — .claude/templates/index.json 을 읽으면 세션 마커(임시 파일)를 기록한다.
  2. PreToolUse(Write|Edit) — 생성/수정 대상 경로가 index의 템플릿 패턴과 매칭되는데 마커가 없으면 permissionDecision: "deny" 로 차단하고, 관련 템플릿을 먼저 탐색하라는 이유를 반환한다.

context7 / code.claude.com 으로 검증한 deny 출력 형식:

{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "관련 템플릿을 먼저 탐색하세요: .claude/templates/index.json"
  }
}
의도된 규칙 위반. 이 하드 차단은 프로젝트 CLAUDE.md의 "훅은 항상 non-blocking" 규칙과 충돌한다. 사용자 승인에 따라 deny를 채택하며, §11 에서 CLAUDE.md를 개정한다.

6규칙 4종

플러그인 rules/ 에 markdown으로 제공. 에이전트가 온디맨드로 로드(developer는 구현 전, ruler는 검증 시). 항상 주입하지 않아 컨텍스트가 가볍다. ruler가 규칙의 소유자다. 대상은 wonderit 팀 고정 스택.

파일내용
backend.mdSpring Boot (Java/Kotlin), 계층 구조(Controller/Service/Repository), JPA/MyBatis 규약
frontend.mdJSP + jQuery + Thymeleaf + 그리드 라이브러리 패턴
security.md입력 검증, 시크릿 관리, 인증/인가, SQL 인젝션·XSS 방지
template-meta.md"템플릿을 위한 템플릿" — 표준 구조·규약: index.json 스키마, 플레이스홀더 표기, 파일 네이밍, 메타데이터 필드

7템플릿 저장소 — 프로젝트-로컬 카탈로그 + 인덱스

플러그인은 범용 Spring/JSP 스캘폴드를 시드로 제공한다. templater가 사용자 코드베이스에서 발굴한 패턴을 카탈로그에 추가하고 index.json 으로 색인한다. 훅은 이 인덱스만 읽어 빠르게 판단한다.

.claude/templates/
├── index.json            # 템플릿 카탈로그 색인 (훅이 읽음)
├── scaffolds/            # 플러그인 시드 + 발굴 스캘폴드
│   ├── spring-module/
│   └── jsp-page/
└── patterns/             # 발굴된 프로젝트-로컬 패턴

index.json 스키마는 template-meta.md 규칙이 정의한다(템플릿 식별자, 매칭 경로 패턴, 설명, 메타데이터).

8요청 문서

플러그인이 빈 양식을 시드로 제공하고, 사용자가 수동으로 작성한다. 커맨드가 검증 후 읽는다.

문서용도위치
create_request.md최초 제작 요청.claude/requests/
modify_request.md대규모 수정 요청.claude/requests/

각 양식은 검증 게이트가 검사할 필수 섹션(목표, 범위, 제약, 수용 기준 등)을 명시한 헤더를 포함한다.

9스킬 3종 변경 없음

이미 plugins/wonder-harness/skills/ 에 존재. 이번 작업에서 신규 개발 대상 아님.

  • grill-me — 계획/설계 심층 인터뷰
  • handoff — 대화 핸드오프 문서화
  • write-a-skill — 신규 스킬 작성

11CLAUDE.md 개정 필수

현재 프로젝트 CLAUDE.md "개발 규칙"의 훅은 항상 non-blocking (차단 아닌 제안) 조항이 §5 deny 훅과 충돌한다. claude-md-management 로 개정: 템플릿 탐색 강제 훅을 명시적 예외로 허용하도록 문구를 수정하고, 신규 컴포넌트(commands/agents/hooks/rules/requests/templates)를 저장소 구조 섹션에 반영한다.

13비목표 / 미결

  • 비목표: 범용 배포(스택 무관) — 본 하네스는 wonderit 팀 고정 스택 전용.
  • 비목표: 스킬 3종 신규 개발 — 이미 존재, 변경 없음.
  • 비목표: 요청 문서 자동 생성(대화형) — 사용자 수동 작성으로 확정.
  • 미결(구현 시 확정): 요청 양식 필수 섹션의 정확한 목록, index.json 필드 세부, 버전 범프 시점(plugin.json + marketplace.json 동시 수정).
